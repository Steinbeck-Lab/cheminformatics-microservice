# Architecture Patterns

**Domain:** Frontend modernization of cheminformatics web UI
**Researched:** 2026-03-12

## Recommended Architecture (Post-Migration)

The migration should preserve the existing structure while adding shadcn/ui's conventions.

```
frontend/
  src/
    App.jsx                    # Router + Layout (unchanged pattern)
    main.jsx                   # Entry point (Vite convention: main.jsx not index.js)
    context/
      AppContext.jsx            # Keep. Update env vars to import.meta.env
    pages/                     # Keep all 9 pages unchanged
    components/
      ui/                      # NEW: shadcn/ui components (auto-generated by CLI)
        button.jsx
        card.jsx
        dialog.jsx
        input.jsx
        select.jsx
        tabs.jsx
        toast.jsx
        tooltip.jsx
        skeleton.jsx
        badge.jsx
        alert.jsx
        command.jsx            # For Cmd+K palette
        sonner.jsx             # Toast provider
        breadcrumb.jsx
        ...
      common/                  # Keep. Gradually replace internals with shadcn/ui components.
      chem/                    # Keep. Update imports only.
      convert/                 # Keep.
      depict/                  # Keep.
      ocsr/                    # Keep.
      tools/                   # Keep.
    services/                  # Keep. Update env var references.
    hooks/                     # Keep.
    utils/                     # Keep.
    lib/
      utils.js                 # NEW: shadcn/ui utility (cn function = clsx + tailwind-merge)
    styles/
      tailwind.css             # REWRITE: @import "tailwindcss" + @theme blocks
      animations.css           # Keep (plain CSS keyframes work in Tailwind v4)
      globals.css              # NEW: shadcn/ui CSS variable definitions for theming
  index.html                   # MOVE from public/ to root (Vite convention)
  vite.config.js               # NEW: Vite configuration
  components.json              # NEW: shadcn/ui CLI configuration
```

### Key Architectural Decisions

**1. Keep the Page/Component/Service separation**

The existing architecture is clean and well-organized. Pages own routing logic, components own UI, services own API calls. Do NOT restructure -- just modernize the internals.

**2. Add `components/ui/` for shadcn/ui**

shadcn/ui components live in `src/components/ui/` by convention. These are source files you own (not node_modules). They use Radix UI primitives underneath. The CLI generates them:

```bash
npx shadcn@latest init      # Creates components.json + lib/utils.js
npx shadcn@latest add button # Generates components/ui/button.jsx
```

**3. Keep AppContext, do NOT add a state library**

The existing context provides: `isDarkMode`, `toggleDarkMode`, `recentMolecules`, `apiConfig`, `isLoading`, `globalError`. This is simple enough for React Context. No need for Zustand/Redux.

**4. The `cn()` utility function**

shadcn/ui uses a utility function that combines `clsx` and `tailwind-merge`:

```javascript
// src/lib/utils.js
import { clsx } from "clsx"
import { twMerge } from "tailwind-merge"

export function cn(...inputs) {
  return twMerge(clsx(inputs))
}
```

This prevents Tailwind class conflicts (e.g., `cn("p-4", condition && "p-2")` correctly resolves to `p-2`).

### Component Boundaries

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| `App.jsx` | Router definition, layout wrapping | All pages via React Router |
| `AppContext` | Global state: theme, recent molecules, API config | All components via useAppContext |
| Pages (`*Page.jsx`) | Route handling, tool tab selection, layout per page | Components via props, services for data |
| View components (`*View.jsx`) | Individual tool UI: form + results display | Services via hooks, AppContext for global state |
| `components/ui/*` | Primitive UI elements (Button, Card, etc.) | Nothing -- pure UI, receive props only |
| `components/common/*` | Shared domain components (SMILESInput, MolFileUpload, etc.) | Services, AppContext, `components/ui/*` |
| Services (`services/*`) | API call layer: Axios requests + response handling | Backend API only |
| Hooks (`hooks/*`) | Reusable logic: API call state, molecule utilities | Services, React state |

### Data Flow

```
User action (click, type, upload)
        |
        v
View component handles event
        |
        v
Calls service function (or hook that calls service)
        |
        v
Service uses Axios instance (services/api.js)
        |
        v
FastAPI backend processes and responds
        |
        v
Service returns data to component
        |
        v
Component updates local state (useState)
        |
        v
React re-renders with new data
```

No changes needed to this data flow. The migration only affects the build system, styling approach, and component primitives.

## Patterns to Follow

### Pattern 1: Gradual Component Replacement

**What:** Replace hand-rolled components with shadcn/ui one at a time, page by page.
**When:** During the shadcn/ui integration phase.
**Why:** Avoids big-bang rewrites. Each page can be tested independently.

```
Step 1: Install shadcn/ui base components (Button, Card, Input)
Step 2: Update Header/Footer/Navigation (shared components)
Step 3: Update one page at a time (start with simplest: AboutPage)
Step 4: Update complex pages last (ChemPage with 14 sub-views)
```

### Pattern 2: CSS Variable Theming

**What:** Use CSS variables for all theme-dependent values. shadcn/ui defines these.
**When:** During dark mode integration.

```css
/* Tailwind v4 approach */
@import "tailwindcss";

@theme {
  /* shadcn/ui defines these CSS variables for you */
  --color-background: hsl(var(--background));
  --color-foreground: hsl(var(--foreground));
  --color-primary: hsl(var(--primary));
  --color-primary-foreground: hsl(var(--primary-foreground));
  /* etc. */
}
```

The existing approach of CSS variables in `tailwind.css` (`:root` and `.dark` blocks) is already the right pattern. shadcn/ui will formalize it.

### Pattern 3: Vite Environment Variables

**What:** Replace CRA's `process.env.REACT_APP_*` with Vite's `import.meta.env.VITE_*`.
**When:** During Vite migration.

```javascript
// Before (CRA)
const API_URL = process.env.REACT_APP_API_URL || "https://...";

// After (Vite)
const API_URL = import.meta.env.VITE_API_URL || "https://...";
```

### Pattern 4: Lazy Loading Routes

**What:** Code-split pages using `React.lazy()` for faster initial load.
**When:** After core migration is stable.

```javascript
const ChemPage = React.lazy(() => import('./pages/ChemPage'));
const ConvertPage = React.lazy(() => import('./pages/ConvertPage'));
// Wrap with <Suspense fallback={<LoadingScreen />}>
```

## Anti-Patterns to Avoid

### Anti-Pattern 1: Big-Bang Migration

**What:** Trying to do Vite + React 19 + Tailwind v4 + shadcn/ui all at once.
**Why bad:** If something breaks, you don't know which change caused it. Debugging is nearly impossible.
**Instead:** One migration per phase. Test thoroughly after each. Commit working states.

### Anti-Pattern 2: Migrating to TypeScript During Stack Upgrade

**What:** Converting .js/.jsx to .ts/.tsx while also changing build tool, framework, and styling.
**Why bad:** TypeScript errors will flood the codebase during migration, making it impossible to verify the other changes are correct.
**Instead:** Keep .js/.jsx. Add TypeScript as a separate future milestone if desired.

### Anti-Pattern 3: Overriding shadcn/ui Defaults

**What:** Heavily customizing every shadcn/ui component immediately after installation.
**Why bad:** Defeats the purpose of a consistent design system. Creates maintenance burden.
**Instead:** Use shadcn/ui defaults first. Only customize after living with the defaults for a while and identifying genuine needs.

### Anti-Pattern 4: Mixing Icon Libraries

**What:** Using lucide-react (shadcn default) + react-icons + FontAwesome in the same project.
**Why bad:** Inconsistent icon sizes, styles, and weights. Larger bundle size.
**Instead:** Pick one icon library and migrate to it. lucide-react for shadcn/ui consistency. If react-icons covers more icon needs, keep react-icons and configure shadcn/ui to use it.

### Anti-Pattern 5: Keeping PostCSS Config with Tailwind v4 Vite Plugin

**What:** Having both `postcss.config.js` and `@tailwindcss/vite` plugin active.
**Why bad:** Duplicate processing. Unexpected behavior. Build errors.
**Instead:** Remove `postcss.config.js` entirely when using `@tailwindcss/vite`.

## Scalability Considerations

This is a small SPA (9 pages, ~55 components) serving a research audience. Scalability is not a primary concern, but here are the considerations:

| Concern | Current (~100 users) | At 1K users | At 10K users |
|---------|---------------------|-------------|--------------|
| Bundle size | ~500KB (CRA, no splitting) | Vite tree-shaking reduces to ~300KB | Add lazy routes for sub-300KB initial load |
| Build time | 30-60s (CRA webpack) | 5-10s (Vite) | Same (Vite scales well) |
| Dev server startup | 10-30s (CRA) | <1s (Vite) | Same |
| CSS size | Tailwind purges unused | Same | Same |
| API calls | Individual per tool | Same | Backend rate limiting handles this |
